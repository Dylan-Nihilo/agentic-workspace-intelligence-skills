## Design Context

### Users

Repository maintainers and engineers use the generated Atlas while learning an unfamiliar codebase. Their job is to move from verified structure to component meaning and then to business paths without reading raw graph JSON or a sprawling dependency diagram.

### Brand Personality

Calm, precise, and evidence-led. The interface should feel like a well-edited engineering atlas: trustworthy enough for investigation, restrained enough for long reading sessions, and explicit about what is known versus unresolved.

### Aesthetic Direction

Use the existing light Swiss engineering-report language: off-white paper, near-black ink, one restrained accent, sharp panels, hairline separators, serif display type, and compact technical metadata. Preserve the top-down progressive canvas. Avoid generic dashboard cards, decorative gradients, dark neon styling, tangled whole-graph views, and repeated visual nodes that obscure shared dependencies.

### Design Principles

1. Reveal one verified layer at a time; every stage must show what changed in the artifact.
2. Prefer one canonical node per repository entity, with multiple relationships converging on it.
3. Start with a readable top-down slice and disclose deeper branches on demand.
4. Distinguish structural facts, Agent-produced semantics, and unresolved limitations visually and textually.
5. Preserve evidence and provenance while keeping the primary surface understandable to a human reader.
