# Human Readable Visual Contract

Use only the four governed Maps, canonical Journey artifacts, and the accepted synthesis narrative as content sources.

## Source boundary

- Read Application Map, Experience Map, Runtime Flow Map and Change Map.
- Read the canonical Journey manifest and its `repo-journey-definition/v1`, `repo-journey-binding/v1`, and closure report refs.
- Read `synthesis/narrative.json` as `repo-synthesis-narrative/v3` for titles, summaries, and declared limitations.
- Preserve entity IDs, step order, branches, binding status, confidence, evidence refs and artifact hashes.
- Do not recover information from source code, raw stores, old architecture/domain/flow views, code-map or prior HTML.
- Fail when a required source or canonical ref/file is missing, stale, schema-invalid, hash-inconsistent, or the Journey Set is not fully closed.

## Information architecture

### Application Map

Show application/bootstrap first, then route-layout-page, component composition, state ownership/data flow, API/client, auth/permission, build/deploy and testing/quality boundaries.

### Experience Map

Organize by audience and goal. Connect entry route/page, interaction, visible feedback and success/failure outcome without inventing missing links.

### Runtime Flow Map

Render each Journey as an ordered sequence. Keep `success`, `failure`, `alternate`, `retry` and `exit` branches visually distinct. Show each binding as `confirmed`, `candidate`, `conflicted` or `missing`.

### Change Map

Start from the selected capability/entity/file. Show reverse dependencies, reachable pages, affected Journeys, state/API/auth surfaces, related tests, build/deploy implications and blocked dimensions.

## Surface

- Use off-white paper, near-black ink and one restrained accent.
- Prefer sharp panels, hairline separators and compact evidence labels.
- Use color plus text/icon labels for state; never rely on color alone.
- Keep paths and evidence refs selectable and traceable.
- Avoid decorative gradients, blobs, fake screenshots, marketing copy and metric-heavy hero sections.
- Keep unresolved semantic, runtime, product-intent and deterministic diagnostic states visible.

## Verification

- The page is nonblank and exposes all four Map sections.
- Runtime Flow order and branches match Journey artifacts exactly.
- Change impacts match Change Map entries exactly.
- Candidate, conflicted, missing and blocked states remain visible.
- Evidence and artifact provenance are reachable.
- Mobile layout does not overlap or truncate essential labels.
- Keyboard navigation, contrast, theme toggle and SVG export work.
- No content is inferred from deprecated public views or generated by the renderer.
